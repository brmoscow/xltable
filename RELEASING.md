# Как правильно коммитить и выпускать версии документации

Документация собирается на [Read the Docs](https://readthedocs.org) (см. `.readthedocs.yaml`).
RTD создаёт отдельную версию документации из каждого **git-тега** и ветки.
Чтобы на сайте работал переключатель версий, релизы нужно тегировать по схеме ниже.

## Схема версий

- Ветка `main` → версия **latest** на RTD (всегда свежая, пересобирается при каждом пуше).
- Git-тег `X.Y.Z` (например `2.0.14`) → версия **X.Y.Z** на RTD (зафиксированный снимок).
- Последний активированный тег → версия **stable** (её стоит сделать версией по умолчанию).

## Порядок выпуска релиза

### 1. Обновить номер версии

Перед релизом обновите `release` в `docs/conf.py`:

```python
release = '2.0.15'
```

### 2. Закоммитить все изменения

Тег фиксирует состояние репозитория, поэтому **сначала** коммитим всё, что должно попасть в релиз (код, доки, номер версии):

```bash
git add -A
git commit -m "2.0.17"
git push origin main
```

Правила для коммитов:

- Рабочее дерево перед тегированием должно быть чистым — `git status` не показывает изменений.
- Сообщение релизного коммита — номер версии (`2.0.15`), как принято в этом репозитории.
- Не тегируйте коммит, если в нём есть незакоммиченные правки документации — они не попадут в версию на сайте.

### 3. Поставить тег и запушить его

Имя тега должно совпадать с `release` из `docs/conf.py`:

```bash
git tag 2.0.15
git push origin 2.0.15
```

> Важно: `git push` **не отправляет теги** автоматически — тег пушится отдельной командой.
> Можно и разом: `git push origin main --tags`.

### 4. Активировать версию на Read the Docs

1. Зайти на readthedocs.org → проект → **Versions**.
2. Найти появившийся тег `2.0.15` и нажать **Activate**.
3. (Один раз) В **Settings → Default version** выбрать `stable`, чтобы посетители по умолчанию видели последний релиз.

Чтобы не активировать вручную каждый раз, настройте автоматизацию:
**Admin → Automation Rules → Add Rule**:

- *Match:* Custom match → `^\d+\.\d+\.\d+$`
- *Version type:* Tag
- *Action:* Activate version

После этого каждый запушенный тег вида `2.0.15` будет активироваться и собираться сам.

## Если нужно исправить документацию старой версии

Тег неизменяем, поэтому правки в уже выпущенной версии делаются так:

1. Исправить доки в `main` и закоммитить.
2. Выпустить новый патч-релиз (`2.0.16`) по схеме выше.

Перемещать существующий тег (`git tag -f`) **нельзя** — RTD и другие клоны репозитория не увидят изменение корректно.

## Шпаргалка

```bash
# 1. обновить release в docs/conf.py
# 2. затем:
git add -A
git commit -m "2.0.15"
git tag 2.0.15
git push origin main --tags
# 3. проверить сборку на readthedocs.org
```
